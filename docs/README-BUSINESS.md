# 잇테이블 (eattable.kr)

### 선주문 / 선결제형 외식 예약 플랫폼

> **"자리가 아니라, 식사 시작을 예약하다"**

---

## 1. 서비스 소개

| 항목 | 내용 |
|------|------|
| **서비스명** | 잇테이블 (EatTable) |
| **도메인** | eattable.kr |
| **한 줄 정의** | 고객이 매장 방문 전에 좌석, 시간, 인원, 메뉴, 결제를 모두 확정하고, 매장은 고객의 도착 예정 상태를 보며 조리와 상차림 타이밍을 조절해 **"예약한 시간에 빠르게 식사를 시작"**하게 만드는 외식 플랫폼 |
| **슬로건** | "자리가 아니라, 식사 시작을 예약하다" |

### 핵심 차별점

캐치테이블이 **자리**를 예약한다면, 잇테이블은 **메뉴와 식사 시작**까지 예약합니다.

기존 예약 앱에서는 착석 후에도 메뉴 선택, 주문, 조리 대기, 결제라는 과정이 남아 있습니다. 잇테이블은 이 모든 단계를 방문 전에 완료해, 고객은 도착 즉시 차려진 음식과 함께 식사를 시작합니다.

## 2. 해결하는 문제

### 기존 예약 앱의 한계

```
[기존 플로우]
예약 확인 → 매장 도착 → 착석 → 메뉴판 확인 → 주문 → 조리 대기(15~30분) → 식사 → 결제

[잇테이블 플로우]
메뉴 선택 + 결제 완료 → 매장 도착 → QR 체크인 → 즉시 식사 시작
```

### 고객이 겪는 문제

| 문제 | 상세 |
|------|------|
| **예약했는데 여전히 기다림** | 좌석만 확보될 뿐, 착석 후 메뉴 선택-주문-조리까지 15~30분 추가 소요 |
| **일행과 메뉴 합의 지연** | 착석 후 메뉴 고르는 시간이 의외로 길고, 특히 단체 모임에서 심각 |
| **결제 과정의 불편** | 식사 후 계산대 대기, 더치페이 정산 번거로움 |
| **시간 압박** | 점심시간 1시간 내에 모든 것을 처리해야 하는 직장인에게 치명적 |

### 점주가 겪는 문제

| 문제 | 상세 |
|------|------|
| **노쇼(No-show)** | 예약 부도로 인한 재료 폐기, 매출 손실. 평균 노쇼율 약 15~20% |
| **주문 응대 과부하** | 피크타임 주문 집중으로 서빙 인력 부족, 고객 불만 증가 |
| **재료 예측 어려움** | 당일 수요를 사전에 알 수 없어 과다/과소 준비 반복 |
| **회전율 저하** | 착석 후 메뉴 결정-조리-결제까지 테이블 점유 시간이 길어짐 |

## 3. 시장 기회

### 외식 예약/주문 시장 핵심 지표

| 지표 | 수치 | 출처/기간 |
|------|------|-----------|
| 예약/웨이팅 앱 월간 활성 사용자(MAU) | **291만 명** | 2022~2025, 185% 성장 |
| 캐치테이블 MAU | **250만 명** | 2025년 기준 |
| 캐치테이블 누적 웨이팅 | **1,710만 건** | 2024년 |
| 테이블오더 누적 결제액 | **10조 원** | 2024년 |
| 스마트오더 도입 시 응대시간 절감 | **평균 10.4분 감소** | 업종별 실증 조사 |

### 시장 공백

```
                    ┌──────────────────────────────┐
                    │         잇테이블 포지션         │
                    │   "예약 + 주문 + 결제 통합"     │
                    └──────────────────────────────┘
                              ▲
               ┌──────────────┼──────────────┐
               │              │              │
        ┌──────┴──────┐ ┌────┴────┐ ┌───────┴───────┐
        │  예약 앱     │ │ 테이블  │ │  배달 앱       │
        │ (캐치테이블) │ │ 오더    │ │ (배민/쿠팡잇츠) │
        │             │ │         │ │               │
        │ 좌석만 예약  │ │ 착석 후  │ │ 매장 방문 없음  │
        │ 메뉴/결제 별도│ │ 주문만   │ │ 외식 경험 없음  │
        └─────────────┘ └─────────┘ └───────────────┘
```

예약 앱은 좌석만, 테이블오더는 착석 후 주문만, 배달 앱은 매장 경험 자체가 없습니다. **방문 전 메뉴 선택부터 결제까지 통합한 플랫폼**은 현재 시장에 부재합니다.

## 4. 타겟 사용자

### 고객 (B2C)

| 우선순위 | 세그먼트 | 핵심 니즈 | 규모 |
|----------|---------|----------|------|
| **1순위** | 직장인 점심/저녁 | 1시간 이내 식사 완료, 대기 시간 제거 | 서울 직장인 약 500만 명 |
| **2순위** | 데이트/약속 | 완벽하게 준비된 식사 경험, 대기 없는 코스 | 20~30대 외식 빈도 월 8회+ |
| **3순위** | 소규모 회식/모임 | 메뉴 사전 합의, 간편 정산, 시간 절약 | 4~8인 단체 외식 수요 |
| **4순위** | 혼자 먹기 어려운 메뉴 고객 | 샤브샤브, 전골 등 2인 이상 메뉴를 함께 먹을 동행 매칭 | 기존 밥동무 서비스 확장 |

### 매장 (B2B)

| 업종 | 적합 이유 |
|------|----------|
| **샤브샤브/전골/찜** | 세트메뉴 기반, 사전 재료 준비가 명확 |
| **고깃집** | 인원별 세트 구성이 표준화되어 있음 |
| **훠궈** | 코스형 주문, 재료 사전 준비 효과 극대화 |
| **코스요리/파인다이닝** | 사전 메뉴 확정이 자연스러운 업종 |
| **한정식/백반** | 상차림 사전 준비로 도착 즉시 제공 가능 |

공통 특징: **세트메뉴 기반으로 사전 준비가 명확한 업종**에서 잇테이블의 효과가 극대화됩니다.

>
> 잇테이블 예약 = "12시에 **삼겹살 2인 세트가 차려져 있습니다**"

## 6. 사용자 여정

### 고객 플로우

```
[1단계: 탐색]          [2단계: 예약]           [3단계: 확정]
매장 검색/추천     →   날짜/시간/인원 선택  →   메뉴 선택 + 결제
                                              (카드/간편결제)

[4단계: 이동]          [5단계: 도착]           [6단계: 식사]
실시간 도착 알림   →   QR 체크인           →   즉시 식사 시작
(매장에 ETA 전달)      (매장: 조리 완료)       (대기 시간 0분)
```

| 단계 | 고객 행동 | 매장 행동 | 시스템 동작 |
|------|----------|----------|------------|
| 매장 선택 | 위치/카테고리/추천으로 매장 탐색 | - | 개인화 추천, 실시간 예약 가능 여부 표시 |
| 메뉴 선택 | 메뉴 구성, 수량, 옵션 확정 | - | 재고/가용 시간대 실시간 검증 |
| 결제 | 카드/간편결제로 선결제 | 예약 접수 알림 수신 | 결제 완료, 예약 확정 |
| 이동 중 | 앱에서 "출발" 버튼 탭 | 도착 예정 시간(ETA) 확인 | GPS 기반 ETA 계산, 매장에 전달 |
| 도착 | QR 체크인 | 조리 완료, 상차림 | 체크인 확인, 매장에 알림 |
| 식사 | 즉시 식사 시작 | 서빙 | 테이블 타이머 시작 |
| 완료 | 리뷰 작성 (선택) | 정산 확인 | 매장 정산 처리 |

### 점주 플로우

```
[예약 접수]         [도착 확인]         [조리 시작]         [제공]           [정산]
주문 내역 확인  →   ETA 모니터링    →   도착 10분 전     →  체크인 확인  →   자동 정산
재료 사전 준비      조리 타이밍 판단     조리 시작            즉시 제공       주간/월간 리포트
```

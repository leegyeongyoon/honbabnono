## 1. 피벗 개요

### AS-IS
- 서비스명: 잇테이블 (혼밥시러)
- 컨셉: 식사 모임 매칭 앱 (밥동무)
- 핵심 기능: 모임 생성/참가, 채팅, 리뷰, 포인트, 밥알지수

### TO-BE
- 서비스명: 잇테이블 (eattable.kr)
- 컨셉: **선주문/선결제형 외식 예약 플랫폼**
- 핵심 기능: 매장 검색, 메뉴 선택, 선결제, 도착 알림, QR 체크인, 매장 조리 관리

### 한 줄 정의
> 고객이 매장 방문 전에 좌석, 시간, 인원, 메뉴, 결제를 모두 확정하고,
> 매장은 고객의 도착 예정 상태를 보며 조리/상차림 타이밍을 조절해
> "예약한 시간에 빠르게 식사를 시작"하게 만드는 외식 플랫폼

### 포지셔닝
- 캐치테이블이 **자리를 예약**한다면, 잇테이블은 **메뉴와 식사 시작까지 예약**한다
- 테이블오더가 **매장 안 주문을 디지털화**했다면, 잇테이블은 그 주문을 **매장 방문 전으로 앞당긴다**

## 4. 타겟 사용자

### 고객 (우선순위)
1. **직장인 점심/저녁** - 시간 부족, 빠른 식사 시작 원함
2. **데이트/약속 고객** - 완벽한 경험, 대기 없는 식사
3. **소규모 회식/모임** - 메뉴 사전 합의, 정산 간소화
4. **혼자 먹기 어려운 메뉴 고객** - 기존 밥동무 기능 유지

### 매장 (초기 타겟)
| 업종 | 적합도 | 이유 |
|------|--------|------|
| 샤브샤브 | 매우 높음 | 인원/세트/상차림 준비가 명확 |
| 전골/찜 | 높음 | 조리 준비 시간 있고 단체 수요 |
| 고깃집 세트 | 높음 | 기본찬/불판/고기 세팅 가능 |
| 훠궈 | 높음 | 세트화 쉽고 준비 단계 분리 가능 |
| 코스요리 | 높음 | 선예약/선결제 문화와 맞음 |

## 8. 도착 기반 조리 타이밍 (핵심 기능)

### 고객 앱 도착 버튼
- 예약 30분 전: 제시간 도착 / 10분 늦어요 / 취소 요청
- 예약 15분 전: 이동 중 / 10분 내 / 5분 내 / 매장 근처
- 도착 시: QR 체크인

### 점주 화면 예시
| 시간 | 고객 | 메뉴 | 도착 상태 | 추천 액션 |
|------|------|------|----------|---------|
| 19:00 | 이OO 2명 | 샤브샤브 2인 | 10분 내 도착 | 육수/상차림 준비 |
| 19:10 | 김OO 4명 | 고기세트 | 10분 지연 | 조리 보류 |

### 원칙

### 예약 상태 머신
```
[pending_payment] ──결제완료──→ [confirmed]
        │                           │
    결제실패                    점주확인+도착예정
        ↓                           ↓
   [cancelled]              [preparing] ──조리완료──→ [ready]
                                                        │
                                                    고객착석
                                                        ↓
                                                   [seated]
                                                        │
                                                    식사완료
                                                        ↓
                                                  [completed]
```

### 서버 모듈
| 모듈 | 재활용 수준 | 전환 방안 |
|------|------------|----------|
| auth | 95% 그대로 | 카카오/이메일 인증 유지 |

| deposits | 50% 대폭 수정 | "약속금" → "선결제" 전환. PortOne 연동 재활용 |

## 13. MVP 범위 (최소 검증)

### 목표
> "정말 사람들이 메뉴까지 선결제하고 예약하는가?"를 검증

### 범위
- 제휴 매장 **3~5곳** (샤브샤브/고깃집 중심)
- 세트 메뉴 **2~3개**/매장
- 고객 예약 → 메뉴 선택 → 결제 → 도착 알림 → QR 체크인
- 점주 예약 확인 → 조리 상태 변경 → 도착 확인

### MVP에서 제외
- POS 연동, GPS 자동 추적, AI 조리 예측
- 모든 메뉴 등록 (세트메뉴만)
- 전국 확장 (1개 상권 집중)
- 복잡한 좌석 최적화

## 16. 사업계획서용 핵심 문구

### 문제 정의
> 기존 외식 예약 서비스는 고객이 원하는 시간에 좌석을 확보하는 데 집중해 왔다. 그러나 실제 외식 경험에서는 착석 이후에도 메뉴 선택, 직원 호출, 주문 접수, 조리 시작, 결제 과정에서 추가 대기 시간이 발생한다.

### 솔루션 정의
> 본 서비스는 사용자가 외식 예약 단계에서 방문 시간, 인원, 메뉴를 선택하고 결제까지 완료할 수 있는 선주문형 외식 예약 플랫폼이다. 매장은 예약 정보를 기반으로 좌석 배정, 재료 준비, 상차림, 조리 타이밍을 사전에 관리할 수 있으며, 고객의 도착 예정 상태를 확인해 음식 품질 저하와 노쇼 리스크를 줄일 수 있다.

### 최종 한 문장
> 이 서비스는 고객이 외식 전에 시간/좌석/메뉴/결제를 확정하고, 매장은 고객의 도착 상태에 맞춰 조리 준비를 조절함으로써, 예약 후에도 발생하는 주문/결제/조리 대기와 노쇼 리스크를 동시에 줄이는 선주문형 외식 예약 플랫폼이다.
